# /handoff-test

최신 스펙의 미완료 todo에서 테스트 작업을 선택해 current_test.md를 작성한다.
AGENTS.md 는 수정하지 않는다.

## 실행 순서

1. `docs/work_flow/specs/` 에서 파일명(NN_ 순번 접두사) 기준 **가장 높은 번호 파일**을 찾는다.
   - 스펙 파일이 없으면 개발자에게 알리고 종료한다.

2. 스펙 파일에서 **미완료 todo** (`[ ]`) 중 `타입: test` 또는 `타입: both`인 항목을 추출한다.
   - 미완료 test 항목을 목록으로 출력하고 개발자에게 어떤 todo를 테스트할지 선택하게 한다.

3. 현재 날짜+시간을 확인한다. `date +%Y%m%d_%H%M`

4. 선택한 todo의 테스트 단계를 분석해 **Codex 검증**과 **개발자 직접 검증**으로 분류한다:

   **Codex 검증** (비대화형 SSH로 실행 가능):
   - 파일 존재 여부 확인 (`ls`, `find`)
   - 문법 검증 (`bash -n`, `python -m py_compile`)
   - SSH 접속·venv 활성화 확인
   - 비대화형 명령 실행 (출력 확인, 에러 여부)
   - `--help` 출력 확인

   **개발자 직접 검증** (대화형 터미널 필요):
   - `input()` 호출이 있는 스크립트 실행 (calibration 등)
   - 물리적 하드웨어 조작이 필요한 단계
   - 육안 확인이 필요한 단계 (모터 추종, 카메라 화각 등)
   - sudo 비밀번호 입력이 필요한 단계

5. 선택한 todo의 내용을 기반으로 `docs/work_flow/context/current_test.md` 를 아래 형식으로 덮어쓴다:

   ```
   # Current Test Target
   <!-- /handoff-test 으로 업데이트. 이전 기록: docs/work_flow/context/history/ -->

   > YYYY-MM-DD HH:MM | 스펙: `docs/work_flow/specs/[파일명]` | TODO: [번호]

   ## 테스트 목표
   [todo 제목 및 테스트 목표]

   ## DOD (완료 조건)
   [todo의 DOD]

   ## 환경
   [예: Orin JP6.2, venv smolvla / DGX Spark]

   ## Codex 검증 (비대화형)
   <!-- Codex가 SSH 비대화형으로 실행하고 결과 컬럼을 채운다 -->
   | # | 단계 | 기대 결과 | 결과 | 비고 |
   |---|---|---|---|---|
   | 1 | | | | |

   ## 개발자 직접 검증 (대화형)
   <!-- 개발자가 Orin Remote SSH 터미널에서 직접 실행하고 결과를 기록한다 -->
   | # | 단계 | 기대 결과 | 결과 | 비고 |
   |---|---|---|---|---|
   | 1 | | | | |
   ```

   - `Codex 검증` 또는 `개발자 직접 검증` 중 해당하지 않는 섹션은 생략한다.
   - test 코드 검증 / prod 코드 검증 구분이 필요한 경우 각 섹션 아래에 소제목으로 추가한다.
   - Codex는 `Codex 검증` 섹션의 결과 컬럼만 채운다. 스펙 파일은 직접 수정하지 않는다.

6. 완료 후 보고:
   - 읽은 스펙 파일 및 선택한 TODO 번호
   - current_test.md 핵심 내용
   - **Codex에게 전달할 프롬프트** (비대화형 검증만 담당)
   - **개발자에게 직접 요청할 대화형 검증 단계** (해당 항목이 있을 경우 명시)
