**발생**: 1건 (본 사이클 전체 구조)

**구체 상황**: 08 spec 은 Phase 1 합의 시점에 Wave 1 (정합 정리 9 todo) 만으로 시작했으나, Wave 1.5 (U 그룹 — interactive_cli audit, 사용자 결정 J) 가 walkthrough 도중 추가됐다. 이후 사용자 결정 K (큰 구조 리팩토링 우선) 로 U2~U5 분해 직전 상태에서 멈췄고, Wave 2 (시연장 셋업) 도 calibrate 단계에서 일시 중단됐다. 결국 원래 16 todo 중 9 todo 만 처리된 상태로 사용자가 "Wave 1 만 한 정합 정리 사이클로 정리하자" 결정하여 미완 요소 전체를 깎아냄.

**영향**: 미완 Wave 2~5 가 다음 spec 으로 이관되고, ad-hoc 5건 흔적 및 U1 산출물·U2~U5 todo 계획 등이 정리됨. spec history 보관 시 실제 처리 범위 대비 원래 계획 범위가 대폭 축소된 형태로 기록됨.

**원인 분석**:
- Phase 1 합의 시점에 Wave 1 (자율 검증 가능 정합 정리) 과 Wave 2~5 (시연장 PHYS_REQUIRED + 학습 + 추론) 를 동일 spec 에 묶음. 두 유형의 환경 의존성 차이가 처음부터 이질적이었음.
- Wave 1.5 (U 그룹) 는 walkthrough 중 ad-hoc 발견으로 추가 — Phase 1 합의에 없던 scope 확장.
- U1 결과 (audit 34건 발견) 후 사용자 결정이 "즉시 처리 vs 구조 리팩토링 우선" 으로 갈렸고, 구조 리팩토링 우선 결정이 Wave 2~5 진입 시점을 불확실하게 만듦.

**07 reflection 비교**: 07 패턴 1 (사용자 walkthrough 가 todo 폭증 트리거) 의 진화형. 07 에서는 walkthrough 가 D 그룹 11건을 추가했고, 08 에서는 walkthrough 가 ad-hoc 5건 + Wave 1.5 추가 + 구조 리팩토링 결정이 이어져 spec 전체 scope 가 제어 불가능 상태로 확장됐다.

**harness 원칙 매핑**:

### 패턴 2 — walkthrough 중 ad-hoc 변경 5건 누적 + 부채화 경계

**발생**: C0·C0b·C0c·C0d·C0e (5건) — 모두 C1 walkthrough 도중 즉석 발생

**구체 상황**: C1 시연장 셋업 walkthrough 중 interactive_cli 코드 문제가 연쇄 발견됐다. 각 발견마다 ad-hoc todo 를 정식 흐름 (task-executor → code-tester → prod-test-runner) 으로 처리했으며, 모두 AUTOMATED_PASS 로 종결했다. 07 reflection 에서 도출한 "walkthrough ad-hoc 변경 처리 정책" (CLAUDE.md §Phase 2) 이 정상 동작한 사례다.

단, 5건이 연속 발생하면서 calibrate walkthrough 가 ~4시간 동안 진행됐고 결국 calibrate 단계에서 세션을 일시 중단했다. 이후 세션에서도 calibrate 재개가 이어지지 않고 Wave 1.5 (U 그룹) 로 방향 전환됐다. 즉 ad-hoc 5건의 흡수는 성공했으나, 이것이 C1 walkthrough 의 *완주 실패* 의 원인이기도 했다.

**현재 룰 검토**:
- CLAUDE.md "walkthrough 중 ad-hoc 변경 처리 정책" (07 reflection 도출) 은 건별 처리 기준을 명시하나, **누적 건수 임계** 는 없음. 5건이 누적될 때 "일단 멈추고 wave 정지 + 정식 todo 분해 검토" 를 orchstrator 가 판단해야 한다는 지침이 없음.
- 각 C0 사건이 발생할 때마다 ad-hoc dispatch → AUTOMATED_PASS → 다음 C1 단계 재시도 → 다시 ad-hoc 발생 의 루프가 반복됐다. 이 루프 자체를 감지하는 메커니즘 없음.
- 5건 누적 후 "미완 interactive_cli 문제가 더 있을 가능성" 을 orchestrator 가 U1 audit 로 연결한 것은 좋은 패턴이나, audit 결과가 Wave 2 진입을 더 복잡하게 만들었다.

**07 reflection 비교**: 07 패턴 2 (ad-hoc 변경의 spec 외부 즉시 적용, 2건) 의 양적 확대. 07 에서는 2건이 검증 이력 없이 적용됐고, 08 에서는 5건이 정식 흐름으로 처리됐으나 walkthrough 완주를 막는 부채로 전환됐다. 정식 흐름 처리가 반드시 walkthrough 완주를 보장하지 않는다는 것이 새로운 학습 신호.

**harness 원칙 매핑**:
- 원칙 9 (최소 차단 병합 게이트) — ad-hoc 누적 건수가 N건을 넘으면 walkthrough 일시 정지 + 전체 점검 (U1 audit 패턴) 으로 전환하는 것이 처리량 관점에서 더 효율적일 수 있음.
- 원칙 3 (에이전트 가독성) — "walkthrough 중 동일 파일군에서 ad-hoc 이 N건 이상 연속 발생하면 전체 코드 품질 점검 전환 권고" 라는 지식이 현재 어디에도 명시되지 않음.

**제안 위치**: 갱신 제안 #2
